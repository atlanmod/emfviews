Get started

Based on the unquestionable success/usefulness of database views to solve similar problems in databases, EMF Views aims to bring the same concept to the modeling world.

Some users only need to see some parts of one model, others have to get the full model extended with data from another model, or simply access to a combination of information coming from different interconnected models. Up to now, creating such “perspectives” transparently in EMF was not really trivial: so let’s introduce you toEMF Views!

Thanks to the three main constructs (inspired from SQL) offered by the tool, designers can create new model views: SELECTing a subset of elements from a model, PROJECTing only some of the properties of those elements and/or JOINing them with elements from other models. A model view is a special type of model whose instances are directly computed at runtime based on the model view definition and concerned actual model(s).

EMF Views, is currently being developed in the context of the TEAP industrial project, by showing different possible applications of model views including:

  • Software architect/developer views relating UML design models and Java code models (cf. Eclipse MoDisco);
  • Enterprise architect views linking (BPMN) business process models, (ReqIF) requirements models and (TOGAF) architecture models;
  • Generate text artifacts from your views (Support for Epsilon Query Language -EGL-);
  • View transformation using dedicated technologies (Support for Atlas Transformation Language -ATL- & Epsilon Transformation Language -ETL-).

Components of EMF Views

Today’s modellers usually work with a set of independent models where each model belongs to one domain or specialized body of knowledge. In order to enable collaboration between several modellers, each modeller must use the same terms to eliminate possible ambiguities that might arise if each one decided to define its own terminology. Therefore, standards, such as UML, BPMN, ReqIf or TOGAF, exist in different domains to define a common set of concepts which are used to guarantee interoperability.

On top of these standards, several tools are built and therefore several models belonging to different domains are common in a modeller’s everyday’s work. In this scenario, occasions arise when useful information might be scattered in models from different domains. In these cases, modelers continue working with their existing models and standards but need to created new artifacts (Traces, Weaving models, Transformations) to exploit the relations between them.

Based on the concepts of virtual views on databases, our approach uses the existing models to create new views on top of them in a non intrusive manner.

Viewtype

A viewtype defines the type of elements that can be present in a view. The contents of a viewtype come from existing concepts defined in one or several existing standards. Therefore the first step in defining a viewtype is to select the standards from which it will be created.

Once the base concepts are selected, a modeller can PROJECT only some of the properties of those concepts. This means that if for security reasons there are attributes and references that are restricted, then the modeller will choose not to project them on the viewtype. In addition to working with projected original concepts, modellers need to JOIN these concepts.

View

Once a model viewtype has been defined, conforming model views can be created. A model view is a special type of model whose instances are directly computed at runtime based on the model viewtype and the concerned actual model(s)..
